For an interview or team presentation, don't start with code.

Start with the **problem**, then the **solution**, then the **architecture**, and only then dive into files.

---

# 1. 30-Second Elevator Pitch

> I built an AI-powered Pull Request Review System using LangGraph and LLMs.
>
> The system automatically analyzes a GitHub repository, understands its architecture, fetches PR changes, reviews each changed file using an LLM, and generates a consolidated code review report.
>
> Instead of reviewing code blindly, the AI first learns the repository context and then reviews the PR with awareness of the project's architecture and design patterns.

---

# 2. Why I Built This

Every team faces these problems:

* PR reviews take time
* Senior developers are busy
* New developers may miss architectural issues
* Existing AI code reviewers don't understand project context

Most AI tools only see:

```python
old code
→
new code
```

They don't know:

* What this repository does
* Existing architecture
* Design decisions
* Folder structure

So I thought:

> What if the AI first studies the repository and then reviews the PR?

That became this project.

# 4. Why LangGraph?

This is usually the first interview question.

Answer:

> I could have written everything in one Python script, but I wanted a workflow-oriented architecture.
>
> LangGraph allows me to model the review process as a graph of nodes.
>
> Each node performs one responsibility and passes state to the next node.
>
> This makes the system modular, scalable, and easy to extend.

---

Explain:

# 11. Why Patch?

Interview question.

Answer:

> Reviewing the entire file is expensive.
>
> GitHub provides only the changed lines through patches.
>
> So the LLM reviews only the code that changed.

Huge token savings.

# 15. Why This Is Better Than One LLM Call

Interviewer will ask this.

Answer:

### Bad Approach

```text
Entire Repo
+
Entire PR
+
Review
```

Problems:

* huge tokens
* expensive
* poor reasoning

---

### My Approach

Step 1:

```text
Understand Repository
```

Step 2:

```text
Review Files
```

Step 3:

```text
Aggregate Findings
```

Advantages:

* modular
* cheaper
* scalable
* easier to maintain

# 16. Is This Multi-Agent?

Current answer:

### Technically

No.

This is primarily a:

```text
LangGraph Workflow
```

with one real agent:

```python
RepositoryAgent
```

and multiple service nodes.

---

### How To Say It

> The current implementation is a workflow-based AI system built using LangGraph.
>
> It contains one specialized Repository Agent and several review services.
>
> The architecture can be extended into a full multi-agent system by introducing dedicated agents for repository understanding, code review, security review, and report generation.

That's a strong answer.
